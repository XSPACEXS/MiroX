# Content Guide — Team & HR Boards

## Overview

Team and HR board content must serve dual audiences: the people being described and the people reading about them. An org chart serves the employee looking for their VP's name AND the executive planning a reorg. A performance review serves the employee understanding their growth AND the manager documenting their assessment. The content must be precise, fair, actionable, and — above all — human.

---

## Essential Sections by Board Type

### Org Chart — Required Sections

| Section            | Purpose                      | Content Type                                       |
| ------------------ | ---------------------------- | -------------------------------------------------- |
| Header             | Company name, chart metadata | Company name, effective date, total headcount      |
| Level 1 (CEO)      | Top of hierarchy             | Role card: name, title, direct reports count       |
| Level 2 (C-Suite)  | Senior leadership            | Role cards with department, headcount, tenure      |
| Level 3 (VP/Dir)   | Middle management            | Role cards with team name, size, specialization    |
| Level 4 (Teams)    | Team-level detail            | Team cards with lead name and member count         |
| Sidebar: Headcount | Aggregate staffing data      | Department breakdown, current/planned/open roles   |
| Sidebar: Metrics   | Organizational health data   | Span of control, manager ratio, tenure, attrition  |
| Sidebar: Flags     | Structural issues to address | Org health warnings with specific recommendations  |
| Dotted Lines       | Matrix reporting             | Cross-functional relationships with context labels |

### Onboarding Board — Required Sections

| Section          | Purpose                | Content Type                                         |
| ---------------- | ---------------------- | ---------------------------------------------------- |
| Header           | Welcome and context    | New hire name, role, team, start date, buddy/mgr     |
| Pre-boarding     | HR-completed prep      | Checkbox items: equipment, accounts, access          |
| Day 1            | First day schedule     | Time-specific agenda with people and locations       |
| Week 1           | Learning phase         | Task checklist focused on observation and learning   |
| Week 2           | Contribution phase     | Task checklist focused on first contributions        |
| Weeks 3-4        | Ownership phase        | Task checklist focused on independent delivery       |
| Key Contacts     | Who to reach for what  | Contact cards with name, role, channel, availability |
| 30-60-90 Goals   | Performance milestones | Milestone goals with success signals per phase       |
| Culture & Values | Cultural orientation   | Company values with practical descriptions           |
| Cultural Norms   | Unwritten rules        | Communication, meeting, PTO, feedback norms          |

### Team Building Board — Required Sections

| Section             | Purpose                      | Content Type                                         |
| ------------------- | ---------------------------- | ---------------------------------------------------- |
| Header              | Session context              | Team name, facilitator, date, location, participants |
| Team Values         | Collective identity          | Individual contributions + voted top values          |
| Strengths Map       | Individual capabilities      | Per-person strength cards + team gap analysis        |
| Working Agreements  | Behavioral commitments       | Numbered rules with specific, actionable language    |
| Fun Facts           | Social bonding               | Personal tidbits and team trivia                     |
| Collaboration Prefs | Work style alignment         | Matrix of focus hours, feedback styles, preferences  |
| Action Items        | Commitments from the session | Owner + due date + specific deliverable per item     |

### Performance Review — Required Sections

| Section           | Purpose                     | Content Type                                          |
| ----------------- | --------------------------- | ----------------------------------------------------- |
| Header            | Review context              | Employee name/role, period, manager, review date      |
| Goals/OKR Review  | Past achievement assessment | Goals with KRs, status badges, evidence per KR        |
| Competency Matrix | Skills assessment           | Competency table with self/manager ratings and gaps   |
| 360 Feedback      | Multi-source input          | Strength themes + growth themes + attributed quotes   |
| Overall Rating    | Summary evaluation          | Rating badge, scale reference, manager narrative      |
| Compensation      | Reward recommendation       | Merit, equity, bonus with approval status             |
| Development Plan  | Forward-looking growth      | Development areas with specific actions and support   |
| Career Path       | Long-term trajectory        | Career ladder, readiness criteria, conversation notes |

## Real Content Examples

### Example 1: Org Chart — Role Card Content

**CEO Card**:

```
Title:     CEO
Name:      Linda Park
Context:   Founded 2018 | Reports to: Board of Directors
Note:      "Linda's direct reports: CTO, CRO, CPO, CFO, VP People (5 reports)"
```

**C-Suite Card**:

```
Title:     CTO
Name:      Marcus Wong
Context:   Engineering | 72 people
Tenure:    Joined 2018 (co-founder)
```

**VP Card**:

```
Title:     VP AI/ML
Name:      Dr. Fatima Al-Hassan
Context:   12 engineers (NEW TEAM)
```

The "NEW TEAM" label is highlighted in amber (#EF6C00) to draw attention to a recently formed team that may need structural attention.

**Team Card**:

```
Name:      Platform Team
Lead:      Ana Kovacs
Size:      8 engineers
```

### Example 2: Onboarding — Week 1 Checklist Content

Each checklist item should be specific enough that the new hire can complete it without asking "what does this mean?"

**Good checklist items**:

- "Complete all mandatory compliance training on Workday Learning (anti-harassment, data security, code of conduct) — due by Friday"
- "Shadow 3 design critique sessions (Mon, Wed, Fri at 2 PM) — observe how the team gives and receives feedback"
- "1:1 meetings with 5 cross-functional partners (30 min each): Ben Torres (Product), Raj Gupta (Backend), Jim O'Brien (Sales), Rachel Adams (CS), David Kim (People)"

**Bad checklist items**:

- "Complete training" (which training? where? by when?)
- "Meet team members" (which ones? how many? what should they talk about?)
- "Set up tools" (which tools? who helps? what access is needed?)

### Example 3: Onboarding — Manager Notes

Manager notes should sound like a real human talking to their new team member. They provide emotional guidance alongside tactical direction.

**Effective manager notes**:

- "Mia's note: Don't worry about being productive on Day 1. Focus on meeting people and absorbing context. The real work starts Week 2."
- "Mia's note: It is totally normal to feel slow this week. Your first design will take 3x longer than it will in 3 months. Focus on learning the system, not speed."
- "By end of Week 4, you should feel comfortable navigating the design system, participating in critiques, and owning a feature from concept to handoff. If not, flag it — it's a signal, not a failure."

**Ineffective manager notes**:

- "Welcome to the team!" (too generic)
- "Please complete all tasks on time." (too formal, creates pressure)
- "Let me know if you need anything." (too vague — new hires do not know what to ask for)

### Example 4: Performance Review — Goal Card with KR Assessment

A goal card must include: the goal itself, each key result, the status badge per KR, and the evidence supporting the assessment.

```
GOAL: Ship the Checkout Redesign

  KR1: Complete user research (5 interviews, 3 usability tests) by Aug 15
  STATUS: EXCEEDED
  EVIDENCE: Completed 7 interviews and 5 usability tests. Research insights
  directly influenced the final design.

  KR2: Ship final designs to engineering by Sep 30
  STATUS: MET
  EVIDENCE: Shipped on Sep 28. No rework needed — engineering praised
  the spec quality.

  KR3: Achieve 15% conversion rate improvement within 60 days of launch
  STATUS: EXCEEDED
  EVIDENCE: Achieved 22% conversion improvement. Best-performing design
  change of H2.
```

### Example 5: Performance Review — 360 Feedback Synthesis

Feedback should be synthesized into themes, then supported by representative quotes. Never dump raw feedback onto the board.

**Strength Theme**:

```
THEME: Exceptional collaboration
QUOTE: "Alex is the designer I most enjoy working with. He comes to sprint
planning prepared, asks smart questions, and never blocks engineering
with late specs."
ATTRIBUTION: Raj Gupta, VP Backend Engineering
```

**Growth Theme**:

```
THEME: Needs to speak up more in strategy conversations
QUOTE: "Alex has great instincts but waits to be asked. I want him to
proactively bring design perspectives to roadmap discussions."
ATTRIBUTION: Aisha Patel, CPO
```

### Example 6: Team Building — Working Agreement

Working agreements must be specific, behavioral, and enforceable. They are commitments, not aspirations.

**Good working agreements**:

- "Design critiques happen every Wednesday at 2 PM. Attendance is non-negotiable. If you can't attend, post your work in #design-critique by noon for async feedback."
- "When you disagree with feedback, say why — not just 'I prefer my version.' Bring rationale: user data, design principles, or technical constraints."
- "No Slack messages after 6 PM or before 8 AM unless it's truly urgent. We respect off-hours. Use scheduled send."

**Bad working agreements**:

- "Be respectful" (too vague — what does respectful look like in practice?)
- "Communicate openly" (how? where? about what?)
- "Do your best work" (not actionable, not measurable, not enforceable)

### Example 7: Headcount Summary Content

The headcount sidebar in an org chart should contain current state, planned state, and delta:

HEADCOUNT SUMMARY (as of March 1, 2026)

### Example 8: Org Health Flags

Org health flags should identify specific structural issues with concrete recommendations:

```
FLAG: AI/ML team has 12 ICs and 1 VP with no middle management layer.
RECOMMENDATION: Hire an Engineering Manager by Q2.

FLAG: Customer Success has 10 CSMs and 1 Director. Span of control (10:1)
exceeds our 8:1 guideline.
RECOMMENDATION: Promote a Senior CSM to Team Lead.

FLAG: Sales team growing 15% but Sales Ops growing 0%. Risk of
operational bottleneck.
RECOMMENDATION: Budget for 2 additional Sales Ops hires in H2.
```

### Example 9: Career Path Content

The career ladder section should show where someone is, where they are going, and what criteria they need to meet:

```
CURRENT: Senior Product Designer (IC4)
  "Alex is here. Strong performer, 2.5 years in role."

NEXT: Staff Product Designer (IC5)
  "Target: H2 2026. Requires: strategic thinking (gap closing),
   scope expansion, cross-team influence."

FUTURE: Principal Designer (IC6) OR Design Manager (M1)
  "Fork in the road. Alex to decide IC track vs. management track
   by mid-2027."

PROMOTION READINESS (for Staff)
| Criteria                           | Status     | Ready?     |
|------------------------------------|------------|------------|
| Independently scopes & leads       | Yes        | Ready      |
| Influences product strategy        | Developing | 6-12 mo    |
| Mentors others effectively         | Yes        | Ready      |
| Recognized by x-functional leaders | Yes (4.8)  | Ready      |
| Delivers outsized business impact  | Yes (22%)  | Ready      |
| Strategic design thinking          | Developing | 6-12 mo    |

SUMMARY: 4 of 6 criteria met. 2 in development. Expected: Q3-Q4 2026.
```

## Copywriting Principles for Team/HR Boards

### 1. Use Names, Not Just Roles

**Bad**: "The CTO reports to the CEO."
**Good**: "Marcus Wong (CTO) reports to Linda Park (CEO)."

Names make org charts human. Roles make them abstract. When someone is new to the organization, names help them remember who is who; roles alone do not.

### 2. Write Checklist Items as Complete Instructions

**Bad**: "Set up tools"
**Good**: "Set up design tools: Figma (team workspace invite), Miro (board access), Jira (design project board)"

Every checklist item should contain enough information to complete the task without additional research.

### 3. Make Feedback Constructive, Not Just Honest

**Bad**: "Alex needs to be more strategic."
**Good**: "Alex has great instincts but waits to be asked. I want him to proactively bring design perspectives to roadmap discussions."

The first version is a label. The second version is a description of the current behavior and a vision of the desired behavior. It gives Alex something specific to work on.

### 4. Quantify Where Possible

**Bad**: "Alex did well on the checkout redesign."
**Good**: "Achieved 22% conversion improvement. Best-performing design change of H2."

Numbers are harder to argue with and easier to remember. They also provide evidence for ratings.

### 5. Signal the Time Horizon

**Bad**: "Improve strategic thinking."
**Good**: "Independently identify and pitch 2 design opportunities to the product roadmap by June 2026."

Every development goal needs a timeline. "Improve X" is an aspiration. "Do Y by Z date" is a plan.

### 6. Include Emotional Context in Onboarding

**Bad**: "Complete 30-day self-assessment form."
**Good**: "Complete 30-day self-assessment form (shared by David Kim, People team) and discuss with Mia. This is a growth conversation, not an evaluation."

New hires are anxious. Every task that could be perceived as evaluative should include context that reduces anxiety.

### 7. Working Agreements Should Sound Like a Team Voice

**Bad**: "Employees shall attend critique sessions as scheduled."
**Good**: "Design critiques happen every Wednesday at 2 PM. Attendance is non-negotiable. If you can't attend, post your work in #design-critique by noon for async feedback."

Working agreements should sound like the team talking to itself — direct, practical, first-person plural ("we") or second-person ("you"). Corporate third-person language kills engagement.

## Common Content Mistakes

### 1. The Generic Org Chart

An org chart with titles but no names, headcounts but no health metrics, and positions but no planned changes. It tells you the architecture but not the story.

**Fix**: Include names at every level. Add headcount delta (current vs planned). Add org health flags. Add open roles. Make the chart a planning tool, not just a diagram.

### 2. The One-Size Onboarding

An onboarding board that is the same for a new engineer and a new designer. The tasks are generic: "Meet the team," "Learn the product," "Complete training."

**Fix**: Create role-specific onboarding boards. An engineer's board includes Git access, CI/CD walkthrough, first PR, and code review. A designer's board includes Figma access, design system review, critique shadowing, and first project brief.

### 3. The Rating Without Evidence

A performance review that says "4.2 / 5.0 — Exceeds Expectations" with no supporting evidence. The number feels arbitrary.

**Fix**: Every rating must be supported by specific evidence from goals, competencies, and feedback. The manager narrative should explain the logic: "Alex achieved a 22% conversion improvement (KR exceeded), was rated 4.8 in cross-functional feedback (highest on team), and has 4 of 8 competencies above level. This warrants an Exceeds Expectations rating."

### 4. The Feedback Dump

A 360-degree feedback section that lists 15 raw quotes with no synthesis. The reader cannot extract meaningful themes.

**Fix**: Synthesize feedback into 3-4 strength themes and 2-3 growth themes. Select 1-2 representative quotes per theme. Include a highlighted "verbatim of note" if one quote captures the overall sentiment particularly well.

### 5. The Aspiration Without Action

A development plan that says "Develop strategic thinking" without any specific actions, resources, or timeline.

**Fix**: Every development area needs 3-5 specific actions ("Shadow Aisha in 3 strategy meetings," "Read The Strategic Designer by David Holston," "Present a design-led opportunity brief at the April product review"), a timeline for each, and support commitments from the manager.

## Content Templates

### Role Card Template

```
Title:     [Role Title]
Name:      [Full Name]
Context:   [Department] | [X] people
Tenure:    Joined [Year] [optional note: co-founder, promoted from X]
```

### Checklist Item Template

```
[ ] [Time/deadline] — [Specific task description] ([who helps] [where it happens] [what tools needed])
```

### Goal/OKR Card Template

```
GOAL: [Goal name — clear, specific, outcome-oriented]

  KR1: [Measurable key result with quantitative target and deadline]
  STATUS: [EXCEEDED / MET / PARTIALLY MET / NOT MET]
  EVIDENCE: [1-2 sentences of specific, factual evidence]

  KR2: [Measurable key result]
  STATUS: [badge]
  EVIDENCE: [evidence]
```

### Feedback Theme Template

```
THEME: [2-5 word theme label]
QUOTE: "[Direct quote from reviewer, 1-3 sentences]"
ATTRIBUTION: [Name, Title] (for manager/self) or ANONYMOUS (for peers/reports)
```

### Working Agreement Template

```
[Number]. [Specific behavior or commitment] [Enforcement mechanism or fallback if violated]
```

### Development Action Template

```
Development Area: [Skill or competency gap]
Goal: [Specific, measurable outcome with timeline]

  Action 1: [Specific activity — what, when, how]
  Action 2: [Specific activity]
  Action 3: [Specific activity]

Support: [What the manager/company commits to providing]
Budget: [Learning stipend, conference, course allocation]
```
