# Mind Maps — Content Guide

## Overview

A mind map's visual design is only as good as the content it contains. This guide covers what to put on mind map boards: the essential sections, the questions they should answer, how to write effective labels and descriptions, and how to ensure the right amount of content — neither sparse nor overwhelming.

---

## Essential Sections Every Mind Map Needs

Regardless of the specific topic, every well-designed mind map board should include these structural elements:

### 1. Central Topic Node

**What it is**: The focal point of the entire map — a concise statement of the subject, question, or challenge being explored.

**Content requirements**:

- A clear, specific topic statement (not vague or generic)
- Ideally phrased as a question ("How might we...?" or "What are the key factors in...?")
- A subtitle with context: date, team name, session purpose, or project phase
- Optional: A brief 1-2 sentence description beneath the topic

**Examples of strong central topics**:

- "How might we reduce new user churn from 35% to 20% by Q3 2026?"
- "Product Launch Strategy: FinanceBot v2.0 — Q2 Planning"
- "Research Synthesis: What do enterprise buyers value most in security tooling?"
- "Competitive Landscape: AI Writing Assistants — March 2026"

**Examples of weak central topics** (avoid these):

- "Brainstorm" (too vague — brainstorm about WHAT?)
- "Ideas" (meaningless without context)
- "Strategy" (which strategy? for whom? when?)
- "Misc Thoughts" (signals lack of purpose)

### 2. Primary Branch Labels

**What they are**: The 4-8 major categories that organize the map's content into meaningful clusters.

**Content requirements**:

- Short, noun-based labels (1-4 words)
- Mutually exclusive categories (no overlap between branches)
- Collectively exhaustive (branches should cover the full scope of the topic)
- Ordered by importance, clockwise from the top (or left-to-right in hemispheric layouts)

**Branch labeling strategies**:

| Strategy          | Description                      | Example                                               |
| ----------------- | -------------------------------- | ----------------------------------------------------- |
| Stakeholder-based | One branch per audience          | Customers, Engineers, Sales, Partners, Executives     |
| Function-based    | One branch per department        | Product, Marketing, Engineering, Design, Finance      |
| Theme-based       | One branch per topic area        | Pricing, Features, Distribution, Support, Competition |
| Time-based        | One branch per period            | Past (Lessons), Present (Status), Future (Goals)      |
| Framework-based   | One branch per framework element | Strengths, Weaknesses, Opportunities, Threats         |
| Question-based    | One branch per key question      | Who?, What?, Why?, How?, When?, Where?                |

### 3. Sub-Branch Content

**What it is**: The actual ideas, insights, data points, and details that flesh out each branch.

**Content requirements**:

- One idea per sticky note or node (atomic content)
- Specific and actionable (not vague generalizations)
- Include evidence, data, or examples where possible
- Tag or color-code items by type (idea, question, decision, risk, action)

**Writing effective sub-branch content**:

Instead of: "Better onboarding"
Write: "Reduce signup form from 8 fields to 3 — 67% of users abandon at step 4"

Instead of: "Social media strategy"
Write: "Launch TikTok presence targeting DevOps engineers — competitors absent from this channel"

Instead of: "Improve pricing"
Write: "Test annual billing discount of 20% — currently losing 40% of monthly users at 6-month mark"

### 4. Cross-Connections

**What they are**: Lines or arrows connecting ideas across different branches, revealing non-obvious relationships.

**Content requirements**:

- Label every cross-connection with the nature of the relationship
- Common relationship types: "enables," "conflicts with," "depends on," "similar to," "opposite of," "builds on"
- Limit to 5-10 cross-connections per map (more becomes unreadable)
- Cross-connections should reveal genuine insights, not obvious relationships

### 5. Legend / Key

**What it is**: A reference guide explaining the board's visual coding system.

**Content requirements**:

- Color key: What each branch color represents
- Shape key: What different shapes mean (if applicable)
- Icon key: What emojis or symbols mean
- Priority system: How importance or voting is indicated
- Status system: What different markers mean (done, in progress, blocked)

### 6. Metadata Zone

**What it is**: Contextual information about the board itself.

**Content requirements**:

- Board title and version number
- Date created and last updated
- Team or individuals who contributed
- Session context (what meeting, what project, what sprint)
- Link to related boards or documents
- Instructions for new viewers (how to read and navigate the map)

## Optional Sections for Advanced Boards

### Action Items Zone

A dedicated section (typically at the bottom or right edge) collecting all decisions and next steps:

- What needs to be done
- Who is responsible
- By when
- Priority level (High / Medium / Low)
- Status (Not Started / In Progress / Done)

**Example**:

```
ACTION ITEMS
1. [HIGH] Hamzeh — Redesign signup form to 3 fields — by March 15
2. [HIGH] Sarah — Run pricing experiment with 20% annual discount — by March 20
3. [MED] Dev team — Prototype TikTok content series — by March 25
4. [LOW] Maria — Research competitor Miro board templates — by April 1
```

### Voting/Priority Indicators

When the mind map is used in collaborative sessions:

- Dot voting results: Numbers or dots next to each item showing vote counts
- Priority badges: "P1," "P2," "P3" labels
- Star ratings: 1-5 stars indicating team consensus on importance
- Heat coloring: High-vote items get warmer colors (red/orange), low-vote items stay cool (blue/gray)

### Evidence and Data Annotations

For research-driven mind maps:

- Source citations: "[Source: Q2 User Survey, n=1,200]"
- Metrics: "Current: 35% churn | Target: 20% churn"
- Confidence level: "High confidence," "Hypothesis," "Needs validation"
- Date of data: "As of March 2026"

### Relationship Matrix

A small table or grid in the corner showing how branches relate to each other:

```
          | Product | Marketing | Engineering | Design |
Product   |    —    |  High     |  High       | Medium |
Marketing |  High   |    —      |  Low        | High   |
Engineering| High   |  Low      |    —        | High   |
Design    | Medium  |  High     |  High       |   —    |
```

### Timeline or Phasing Overlay

When ideas need to be sequenced:

- Phase 1 / Now: Items for immediate action (marked with a "NOW" badge)
- Phase 2 / Next: Items for next quarter (marked with "NEXT")
- Phase 3 / Later: Items for future consideration (marked with "LATER")

## What Questions Should the Board Answer?

A well-built mind map should allow any viewer to quickly answer:

### Primary Questions (Answerable in <30 seconds)

1. What is the main topic or question being explored?
2. What are the major categories or themes?
3. How many ideas or items are there in total?
4. Which areas have the most content (and thus the most attention)?

### Secondary Questions (Answerable in 1-3 minutes)

5. What are the top 3-5 most important ideas?
6. What connections exist between different branches?
7. Who contributed or is responsible for what?
8. What decisions have been made?

### Deep Questions (Answerable in 5-10 minutes)

9. What evidence supports the key ideas?
10. What gaps exist — which branches are underdeveloped?
11. What conflicts or tensions exist between ideas?
12. What are the specific next steps and who owns them?

## Data and Metrics to Include

### Quantitative Elements

Mind maps are primarily qualitative, but strategic inclusion of data makes them more compelling:

- **Benchmarks**: "Industry average: 12% conversion, Our current: 8%"
- **Goals/Targets**: "Q3 Target: 15% conversion"
- **Trends**: "Up 3% MoM" or "Declining since January"
- **Comparisons**: "Competitor A: $49/mo, Us: $39/mo, Competitor B: $59/mo"
- **Impact estimates**: "Est. revenue impact: $120K/year"
- **User data**: "Reported by 67% of enterprise accounts in Q1 survey"

### Where to Place Data

- **On sticky notes**: Brief metrics alongside the idea ("Reduce churn — currently 35%, target 20%")
- **As annotations**: Small text below or beside the relevant node
- **In a dedicated data zone**: A small section of the board with key metrics that multiple branches reference
- **On connectors**: "This idea could impact that metric by an estimated 5%"

## Real-World Content Examples

### Example 1: Product Feature Brainstorm

**Central Topic**: "What features should we build for FinanceBot v2.0?"

**Branch: User Experience**

- "One-click expense reporting — users currently take 7 clicks"
- "Mobile-first receipt scanning with OCR — 60% of expenses happen on-the-go"
- "Smart categorization using AI — manually categorizing costs 15 min/week per user"
- "Dashboard customization — top 3 request in feedback portal since Jan"

**Branch: Integration**

- "QuickBooks bi-directional sync — 78% of target customers use QB"
- "Slack notifications for expense approvals — reduce approval time from 48h to 4h"
- "Bank feed auto-import — eliminate manual data entry entirely"
- "SSO with Okta/Azure AD — required for enterprise tier"

### Example 2: Marketing Campaign Brainstorm

**Central Topic**: "How do we launch the 2026 Summer Campaign for CloudSync Pro?"

**Branch: Channels**

- "LinkedIn sponsored content targeting VP Engineering — CPL historically $45"
- "YouTube product demo series (5 episodes) — competitor CloudBase got 500K views with similar"
- "Partner webinar with AWS — joint audience of 25K qualified leads"
- "Product Hunt re-launch — time with v3.0 feature release"

**Branch: Messaging**

- "Lead with security compliance angle — SOC2 Type II just achieved"
- "Customer proof points: Acme Corp saved 120 engineering hours/month"
- "Competitive displacement: 'Switch from LegacySync in 15 minutes' campaign"
- "Developer-first messaging: 'The API your engineers actually want to use'"

### Example 3: Research Synthesis

**Central Topic**: "What do users need from our documentation?"

**Branch: Pain Points**

- "Cannot find answers to specific questions — search is broken (mentioned by 23/30 interviewees)"
- "Code examples are outdated — reference v1.x when current is v3.2"
- "No clear getting started path — users spend 45+ min before first API call"
- "Error messages don't link to troubleshooting guides"

**Branch: What Works**

- "Interactive tutorials get 4.5/5 satisfaction — best-rated content type"
- "API reference completeness praised by experienced developers"
- "Community-contributed examples on GitHub have higher engagement than official docs"
- "Video walkthroughs preferred by non-developer personas (PMs, designers)"

## Copywriting Guidelines

### The One-Idea-Per-Sticky Rule

Each sticky note should contain exactly one atomic idea. If a sticky note contains "and" or lists multiple points, it should be split into separate stickies. This rule enables:

- Independent voting on each idea
- Easy reorganization (move one idea without moving others)
- Clear attribution (who suggested what)
- Accurate counting (how many ideas per branch)

### Writing Style

**Active voice**: "Redesign the signup form" not "The signup form should be redesigned"
**Specific nouns**: "Enterprise accounts with 50+ seats" not "Big customers"
**Quantified claims**: "Reduce load time from 3.2s to under 1s" not "Make it faster"
**Action verbs**: "Launch," "Test," "Measure," "Build," "Interview" not "Look into," "Think about," "Consider"

### Headline Writing for Branch Labels

Branch labels are headlines. They should be:

- **Scannable**: Readable in under 1 second
- **Parallel**: All branches use the same grammatical structure (all nouns, all verbs, all questions)
- **Distinct**: No two branches could be confused for each other
- **Complete**: The full set of branches covers the topic without obvious gaps

Good parallel structure:

- "Acquisition | Activation | Retention | Revenue | Referral" (AARRR framework — all nouns)
- "Who buys? | Why now? | What blocks? | How to reach? | What to say?" (all questions)

Bad parallel structure:

- "Marketing | How to improve engineering | Customer feedback | Revenue" (mixed nouns, questions, and concepts)

## Content Volume Guidelines

### How Much is Too Much?

A mind map loses its power when it becomes an exhaustive document. The sweet spot:

| Map Type                  | Total Items | Items per Branch | Max Depth  |
| ------------------------- | ----------- | ---------------- | ---------- |
| Quick brainstorm (15 min) | 15-30       | 3-6              | 2 levels   |
| Standard session (30 min) | 30-60       | 5-10             | 3 levels   |
| Deep exploration (1 hr+)  | 60-120      | 10-20            | 3-4 levels |
| Research synthesis        | 40-80       | 8-15             | 3 levels   |

**Warning signs of too much content**:

- Sticky notes overlap or are jammed against each other
- Connector lines create a tangled web
- You need to scroll more than 3 screen-widths to see the full map
- Branch labels are lost in the sea of sub-items
- The central topic is no longer the visual focal point

**Warning signs of too little content**:

- Branches have only 1-2 items (feels underdeveloped)
- Large areas of empty space within branch zones
- The map could be replaced by a simple bullet list
- No cross-connections exist (ideas are not interacting)

### The Goldilocks Principle

For each branch: **5-8 items is the sweet spot.** This provides enough variety to be interesting, enough specificity to be actionable, and not so much that it becomes a data dump. If a branch has more than 12 items, consider splitting it into two branches. If a branch has fewer than 3 items, consider merging it with a related branch.

## Data Visualization Within Mind Maps

### When to Use Charts

Mind maps are primarily text-and-shape compositions, but small inline visualizations can add powerful context:

**Small pie charts** (80-120px diameter):

- Show market share, team allocation, or budget distribution
- Place next to the branch heading they describe
- Limit to 3-5 segments (more becomes unreadable at this size)

**Mini bar charts** (150-200px wide, 80-100px tall):

- Compare 3-5 values (e.g., competitor pricing, feature scores)
- Place within the relevant branch zone
- Use branch color for the primary bar, gray for comparison bars

**Sparklines** (100-150px wide, 30-40px tall):

- Show trends over time (growth, decline, seasonal patterns)
- Place as annotations beneath relevant ideas
- No axis labels needed — the shape of the line tells the story

### When to Use Tables

Small comparison tables work well within mind maps:

- Feature comparison (3-5 features x 2-4 options)
- Scoring matrices (criteria x options)
- Status trackers (item x status)

**Table guidelines**:

- Maximum 5 rows x 5 columns (anything larger belongs in a separate document)
- Use color fills in cells (green/yellow/red or branch colors)
- Header row should be bold with a distinct background
- Place tables in their relevant branch zone, not floating

### When to Use Lists

Simple numbered or bulleted lists within a frame:

- Prioritized lists (top 5 features, top 3 risks)
- Step sequences (1. Research, 2. Prototype, 3. Test)
- Checklists (items with checkboxes)

**List guidelines**:

- Maximum 7-10 items per list
- Use consistent formatting (all numbered, or all bulleted)
- Place within a subtle frame to distinguish from the organic sticky notes

## Content Templates by Mind Map Type

### Brainstorming Board Content Template

```
CENTER: "How might we [specific challenge]? — [Team] / [Date]"

BRANCHES (adapt to context):
1. Customer Perspective — ideas from the user's point of view
2. Technical Solutions — engineering and architecture ideas
3. Quick Wins — ideas implementable in <2 weeks
4. Moonshots — bold ideas that could be game-changers
5. Process Improvements — workflow and methodology ideas
6. External Inspiration — ideas borrowed from other industries

EACH BRANCH: 5-10 specific, actionable ideas with data support
CROSS-CONNECTIONS: 3-5 links between branches
ACTION ZONE: Top 5 ideas with owners and deadlines
LEGEND: Color key + voting results + priority markers
```

### Concept Map Content Template

```
CENTER: "[Concept Name] — Understanding [domain]"

BRANCHES (hierarchical):
1. Definition — what it is, formal and informal
2. Components — what it consists of
3. Relationships — how it connects to other concepts
4. Examples — real-world instances
5. Applications — how it is used in practice
6. Common Misconceptions — what it is NOT

EACH BRANCH: Mix of definitions, examples, and explanations
CROSS-CONNECTIONS: Labeled with relationship types
LEGEND: Shape key (concepts vs examples vs definitions)
```

### Idea Exploration Content Template

```
CENTER: "[Specific Idea] — Deep Dive Exploration"

BRANCHES:
1. The Idea — detailed description and vision
2. Target User — who benefits and how
3. Feasibility — technical, financial, organizational
4. Impact — expected outcomes and metrics
5. Risks — what could go wrong
6. Next Steps — validation experiments and actions

EACH BRANCH: Evidence-based analysis with specific data
CROSS-CONNECTIONS: Risk-to-mitigation links, impact-to-metric links
DECISION ZONE: Go / No-Go / Need More Info with criteria
```

### Topic Clustering Content Template

```
CENTER: "[Research Question or Theme] — Cluster Analysis"

CLUSTERS (emerge from data):
1. [Theme A] — 8-15 related items grouped by affinity
2. [Theme B] — items sharing a common thread
3. [Theme C] — distinct cluster of related concepts
4. [Theme D] — another thematic group
5. [Outliers] — items that don't fit neatly (important to keep)

EACH CLUSTER: Items with source attribution
CROSS-CLUSTER CONNECTIONS: Labeled relationships
INSIGHTS ZONE: Top 3-5 patterns observed across clusters
META: Methodology (how clustering was done), participant list, date
```
